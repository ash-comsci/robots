1. FTC at a Glance

FIRST Tech Challenge (FTC) is a robotics program where teams design, build, program, and drive a robot to play a themed game on a 12 ft × 12 ft field. Each season introduces a new game with unique scoring tasks.

Match Structure (Typical)

  • Autonomous (~30s): Robot runs pre‑programmed routines.
  • Tele‑Op (~1:30): Drivers control the robot and score.
  • Endgame (~30s): Special scoring opportunities; precision matters.

Always check the current season’s Game Manuals for exact rules, scoring, and constraints.

Core Team Roles

  • Builder: Mechanical design, assembly, reliability.
  • Programmer: TeleOp control + autonomous routines.
  • Outreach/Marketing: Engineering Portfolio, community engagement, judging.

2. Safety & Lab Practices

  • Safety glasses anytime cutting/drilling/testing spinning parts.
  • Hair tied back; no loose clothing/jewelry; closed‑toe shoes.
  • Use the right tools, correct hex sizes, and proper torque.
  • Power off before wiring; avoid battery shorts and pinched wires.
  • Use only FTC‑approved batteries/chargers; never use damaged batteries.
  • Keep the pit clean: sorted parts, clear test zone.
  • Two‑person rule for risky operations: one operates, one spots.

3. Studica FTC Starter Kit Overview

The Studica kit includes structure, motion components, and fasteners to build a robust starter bot.

Typical Contents (May Vary)

  • U‑channel, plates, brackets, gussets.
  • Wheels/hubs (traction/omni), shafts/axles, bearings.
  • Gears/belts/chains for power transmission.
  • DC motors + mounts; standard servos.
  • Fasteners: screws, locknuts, spacers, standoffs, collars.
  • Basic tools: hex keys, wrench; add drivers, snips, threadlocker.

Organizing Your Kit

  • Bins: structure, motion, fasteners, electronics, tools.
  • Simple BOM spreadsheet for inventory.
  • Prototype bin separate from competition‑ready parts.

4.1 Builder

Main Goal: Assemble a durable, serviceable robot that scores reliably.

Starter Path (Weeks 1–3)

  1. Assemble square chassis; check alignment.
  2. Install 4‑wheel tank drive; keep CG low.
  3. Mount control hub + battery; plan wire paths.
  4. Add one simple scoring mechanism.
  5. Add guards to protect wires/gears.

Core Practices

  • Design before build (sketch/whiteboard).
  • Measure twice; dry‑fit, then tighten.
  • Square & symmetric; equal spacers.
  • Blue threadlocker on metal‑metal; avoid plastics.
  • Aligned shafts & proper tension.
  • Serviceability: easy access to spares.
  • Keep weight low & centered.

Mechanism Ideas (Start with One)

  • Intake + hopper
  • Simple arm + servo claw
  • 2–3 stage linear slide
  • Passive drop‑off chute

Builder Meeting Checklist

  • Review goals & sketch/CAD.
  • Check fasteners & threadlocker.
  • Verify wheel alignment & belt/chain tension.
  • Motion smooth through full range.
  • Wires clear and strain‑relieved.
  • Log changes with measurements.

4.2 Programmer

Main Goal: Reliable TeleOp first, then consistent autonomous.

Tooling Options

  • Blocks (OnBot): great for first‑timers.
  • Java (OnBot/Android Studio): more control.
  • Driver Station: Driver Hub or Android phone.

Bring‑Up Steps

  1. Wire correctly; secure connectors.
  2. Create/Name Robot Config; map ports.
  3. Minimal TeleOp drive + mechanism buttons.
  4. Telemetry: voltage, encoders, sensors.
  5. Safety interlocks (limits/soft stops).

From TeleOp to Autonomous

  • Encoders for distance/turns.
  • IMU for heading hold/precise turns.
  • Distance/Color sensors if available.
  • State machines with timeouts.
  • (Optional) field‑centric control.

Structure Tips

  • Subsystem classes (drive/arm/intake).
  • Centralize constants & ports.
  • Telemetry & tuning logs.
  • Version control & backups.

Programmer Meeting Checklist

  • Config matches wiring.
  • Drive base straight & turns consistent.
  • Test mechanisms individually.
  • Record tuning values & notes.
  • Backup after major changes.

4.3 Outreach / Marketing

Main Goal: Tell the story, engage the community, and prepare a clear, concise Engineering Portfolio.

Engineering Portfolio Essentials

  • Audience: judges (~15 pages; verify current rules).
  • Include: team, design process, code highlights, outreach impact, lessons, future plans.
  • Skimmable: headings, diagrams/photos, charts, callouts.

Notebook & Documentation

  • Log every meeting: goals, tasks, outcomes, next steps.
  • Photos of prototypes, code screenshots, test data.
  • Outreach log: date, audience, impact, reflections.

Community & Sponsors

  • One‑page team handout for schools/sponsors.
  • Plan 2–3 events (demos/workshops).
  • Send thank‑yous with photos & metrics.

Judging & Pit Prep

  • 5‑minute presentation: problem → design → results → impact.
  • Welcoming pit visuals; tidy & ready.
  • Everyone can explain their role & one learning.

5. Project Management & Season Timeline

Adopt a simple rhythm: stand‑up, focused work blocks, debrief.

Roles & Routines

  • Stand‑Up (5–10m): Yesterday → Today → Blockers.
  • Work Blocks (60–90m): Subteams in parallel, sync often.
  • Demo & Debrief (10–15m): Show progress, record data, set next steps.

Suggested 12‑Week Plan (high‑level)

  1. Onboarding, safety, kit inventory, goals, game analysis.
  2. Chassis prototype, control bench test, portfolio outline.
  3. Drivetrain driving; simple mechanism; first TeleOp.
  4. Mechanism V1 mounted; auto framework; outreach plan.
  5. Testing/tuning; reliability fixes; portfolio v1 pass.
  6. Mechanism V2; autonomous path v1; outreach event.
  7. Driver practice; spares list; portfolio refine.
  8. Full match sims; auto consistent; outreach event.
  9. Inspection checklist; pit layout; judge draft.
  10. Scrimmage; bug fixes; portfolio final draft.
  11. Final practice; pack/label; mock judging.
  12. Comp week: inspections, matches, judging, reflection.

6. Electrical & Wiring Essentials

  • Mount hubs with access to ports & LEDs.
  • Use FTC 12V battery + main switch; secure with straps.
  • Label wires both ends; print the port map.
  • Strain relief: route along structure, avoid pinch points.
  • Use ferrules/crimps; tug‑test every connection.
  • Separate power & signal where possible.
  • Carry spare fuses, battery, and critical cables.

Example Port Map

Motors: M0 LF, M1 RF, M2 LR, M3 RR, M4 Arm/Slide
Servos: S0 Claw, S1 Wrist
Sensors: I2C IMU, Analog Distance, Digital Limit (upper/lower)

7. Testing & Iteration

  • Write a test plan before changes (trials, criteria).
  • Use telemetry: speed, cycle time, errors, voltage.
  • Change one variable at a time; log results.
  • Regression checklists after updates.
  • Video tests; analyze failures in slow‑mo.

Sample Goals

  • Intake: 5 game pieces < 20s, no jams.
  • Arm/Slide: pickup → score < 2s, < 5° overshoot.
  • Drive: 3m straight < 5cm drift; 90° turn within ±3°.

8. Competition Day Playbook

Pre‑Event Packing

  • Robot + spare battery + charger.
  • Driver Hub/phones, gamepads, cables, power strip.
  • Toolkit + spares: wheels/hubs/shafts/belts/chains/links/fuses/servos/motor/cables.
  • Docs: portfolio, outreach log, port map, inspection checklist.
  • Pit: poster/tri‑fold, tape, markers, sanitizer, snacks, water.

Inspections

  • Hardware: size/weight, secure wiring, no sharp edges.
  • Field: connect to control, basic comms/drive test.
  • Software: team # correct, OpModes clear, no unsafe behaviors.

Drive Team Flow

  • Roles: Driver, Operator, Coach.
  • Pre‑match: alliance plan, battery check, reset encoders, set start config.
  • During: drive clean, avoid fouls; consistency over risk.
  • Post: power down, damage check, battery swap, log issues.

Judging

  • Arrive early; everyone speaks to their contributions.
  • Show data: charts, failures, design changes from feedback.
  • Gracious Professionalism: help, share, respect.

5A. 8‑Week Detailed Schedule by Role

Use these weekly checklists and hand‑offs to coordinate Builders, Programmers, and Outreach.

Week 1

Builder

  • Inventory kit; set labeled bins.
  • Safety training; workspace rules.
  • Assemble square chassis & 4‑wheel drive.

Programmer

  • Install SDK/tools; update hub firmware.
  • Create Robot Config; map motors.
  • Minimal TeleOp + basic telemetry.

Outreach

  • Branding (name/logo/colors), photo folder.
  • Notebook: bios, goals, safety policy.
  • Document unboxing + chassis build.
Dependencies
  • Programmer needs mounted hub + known ports from Builder.
  • Outreach needs photos/screenshots from both subteams.
Week 2

Builder

  • Secure battery/hubs; plan strain‑relief.
  • Drivetrain test (straight & turns); align.
  • Prototype scoring mechanism.

Programmer

  • Refine TeleOp mapping (driver/operator).
  • Encoder helpers (drive straight/turn angle).
  • Autonomous framework skeleton.

Outreach

  • Meeting logs + first social update.
  • Collect portfolio references.
Dependencies
  • TeleOp refinement depends on stable drivetrain tests.
  • Mechanism timing informs programming integration.
Week 3

Builder

  • Mount mechanism with hard stops & guards.
  • Check full motion; verify CG.

Programmer

  • Integrate mechanism control + safety limits.
  • Validate encoder distance/angle on tiles.
  • Record baseline scoring cycle time.

Outreach

  • Document choices/trade‑offs + test data.
  • Plan first outreach demo.
  • Draft portfolio structure.
Dependencies
  • Mechanism mounting complete before code integration.
  • Outreach needs test data for portfolio.
Week 4

Builder

  • Mechanism V2 for repeatability.
  • 10 repeated scoring cycles; label wires.

Programmer

  • IMU heading hold + precise turns.
  • Autonomous V1 (park/score) with timeouts.
  • Full match simulation.

Outreach

  • Run outreach demo; collect metrics.
  • Portfolio updates + charts.
  • Start sponsor outreach.
Dependencies
  • Auto accuracy depends on alignment & consistency.
  • Outreach content uses charts & sim video.
Week 5

Builder

  • Stress tests; reinforce weak points.
  • Spare parts checklist.

Programmer

  • Error handling & interlocks.
  • Auto tuning (speed vs reliability).
  • Tag known‑good backup.

Outreach

  • Portfolio Draft V1.
  • Judge deck (5‑min) build.
  • Share stress‑test results.
Dependencies
  • Programmer tuning follows reliability fixes.
  • Judge deck pulls finalized photos/data.
Week 6

Builder

  • Quick repair practice.
  • Inspection checklist dry‑run.
  • Pit tools finalized.

Programmer

  • Scrimmages with metrics.
  • Telemetry dashboards.
  • Judging tech notes.

Outreach

  • Portfolio V2 polish.
  • Judge presentation practice.
  • Sponsor updates sent.
Dependencies
  • Drive practice requires stable hardware/code.
  • Outreach rehearsals need consistent metrics.
Week 7

Builder

  • Pack competition kit; label/charge batteries.
  • Final calibration; pit repair checklist.

Programmer

  • Code freeze; USB/cloud backups.
  • Verify firmware/configs; print port map.
  • Assist pit materials.

Outreach

  • Print portfolio/notebook.
  • Finalize logistics.
  • Pre‑write event posts.
Dependencies
  • Accurate printed docs (port map, checklists, portfolio) for all.
  • Code freeze means no mapping changes in hardware.
Week 8 (Competition Week)

Builder

  • Pass inspections; fast between‑match repairs.
  • Monitor wear; swap as needed.

Programmer

  • Field‑tune auto starts; safe timeouts.
  • Monitor telemetry; small reliability fixes only.

Outreach

  • Present portfolio; everyone speaks.
  • Capture media; thank sponsors.
  • Collect lessons for debrief.
Dependencies
  • Clean hand‑offs: battery swaps, resets, OpMode selection.
  • Judging narrative needs consistent data & clear roles.
